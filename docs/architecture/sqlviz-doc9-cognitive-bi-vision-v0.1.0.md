## The Doctoral Thesis

### Full Title

> **"Cognitive Compilation of Business Intelligence Dashboards through
> Knowledge Graphs, Multi-Objective Optimization and Cognitive Digital Twins"**

### Abstract

Current Business Intelligence systems operate as passive rendering engines:
they receive data and produce visualizations, but they do not understand
what the user is trying to learn, how much of the business they have covered,
whether two panels answer the same question, or whether the user will
actually read the information presented.

This thesis proposes a new paradigm — Cognitive Dashboard Compilation —
in which a BI system:

1. Constructs a **Knowledge Graph** of business entities and metrics
   from SQL queries, treating each query as a knowledge-acquisition event
2. Computes **Information Gain** per panel to detect redundancy and guide
   the user toward high-value analytical questions
3. Implements a **Curiosity Engine** that proposes the next analytical
   question based on the current state of knowledge
4. Measures **Knowledge Coverage** — what portion of the business domain
   is analytically covered vs. analytically blind
5. Quantifies **Analytical Uncertainty** per panel and communicates
   confidence explicitly to the user
6. Models **Cognitive Cost** of each panel and minimizes the total
   cognitive load of the dashboard
7. Simulates **Attention Patterns** using visual salience models to
   optimize panel ordering and layout
8. Assigns each dashboard a **Genome** — a compact descriptor of its
   analytical structure enabling cross-dashboard comparison
9. Enables **Dashboard Evolution** through learning of user reasoning patterns
10. Builds a **Cognitive Digital Twin** of the user's analytical reasoning,
    not just their preferences

The system is grounded in information theory (Shannon entropy, mutual
information), cognitive load theory (Sweller 1988, Miller 1956),
visual perception research (Cleveland & McGill 1984, Healey & Enns 2012),
and knowledge graph literature (Hogan et al. 2021).

### Research Questions

```
RQ1: Can the information-theoretic content of a SQL query be used to
     construct a knowledge graph that tracks business domain coverage?

RQ2: Is it possible to detect analytical redundancy between dashboard
     panels using mutual information measures on the underlying query sets?

RQ3: Can a curiosity engine generate analytically valid next-step
     recommendations that improve decision-making outcomes?

RQ4: Does explicit uncertainty quantification per panel improve user
     trust calibration compared to confidence-implicit BI displays?

RQ5: Can cognitive cost be modeled sufficiently well to optimize
     panel ordering and reduce measurable cognitive load?

RQ6: Can a Cognitive Digital Twin predict dashboard preferences with
     higher accuracy than role-based personalization?
```

### Hypothesis

```
H1: A dashboard generated by Cognitive Dashboard Compilation
    achieves higher information coverage with lower cognitive cost
    than a manually assembled dashboard of the same size.

H2: Users navigating a Curiosity Engine-guided analytical session
    discover statistically significant insights with fewer SQL queries
    than users navigating without guidance.

H3: Cognitive Digital Twins trained on behavioral interaction data
    predict individual dashboard preferences with higher precision
    than demographic or role-based segmentation.
```

### CompassQL (Wongsuphasawat et al., 2016)

**Paper:** "Towards a General-Purpose Query Language for Data Visualization"
**Origin:** Tableau Research / University of Washington

**What it does:**

```
CompassQL is a query language for visualization recommendation.
The user specifies a partial visualization spec (wildcard fields)
and CompassQL enumerates and ranks candidate completions.

Input:   Partial VegaLite spec with wildcards
Output:  Ranked list of candidate visualizations

Example:
    {mark: ?, encoding: {x: {field: "date"}, y: {field: "sales"}}}
    → ranks: line (0.91), bar (0.83), area (0.72)
```

**Key contribution:**
- Systematic enumeration of visual design space
- Perceptual effectiveness scoring (based on Cleveland & McGill)
- Foundation for Voyager (see below)

**What CompassQL does NOT do:**

```
❌ Does not understand SQL — works on pre-aggregated data specs
❌ Does not model the user's analytical intent
❌ Does not track knowledge coverage across queries
❌ Does not generate the NEXT question to ask
❌ No concept of information gain, redundancy, or cognitive cost
❌ No knowledge graph of business entities
```

**How CBIOS goes further:**
SQLviz starts from SQL (not from a partial visualization spec)
and adds all the layers that CompassQL does not address: intent,
coverage, information gain, uncertainty, cognitive cost, and curiosity.

### Draco (Moritz et al., 2019)

**Paper:** "Formalizing Visualization Design Knowledge as Constraints"
**Origin:** University of Washington / Tableau Research

**What it does:**

```
Draco encodes visualization design guidelines as Answer Set
Programming (ASP) constraints. It generates visualizations that
satisfy hard constraints and minimize soft constraint violations.

Architecture:
    User partial spec
        → ASP solver (clingo)
        → Visualization satisfying constraints
        → Ranked by weighted soft-constraint violations

Core insight: Visualization design rules are constraints,
not lookup tables. The same insight drives V0.2 ConstraintEngine (DOC 5 §18.1).
```

**Key contribution:**
- First system to treat visualization as constraint satisfaction
- Formal grounding of design rules from academic literature
- Auditable: every decision traces to a named rule

**What Draco does NOT do:**

```
❌ No SQL parsing — starts from pre-specified data fields
❌ No intent classification — does not understand WHY the user queries
❌ No knowledge graph construction
❌ No information gain or redundancy detection
❌ No dashboard-level reasoning — focuses on single chart selection
❌ No cognitive modeling (cost, attention, uncertainty)
```

**Direct connection to SQLviz V0.2:**
DOC 5 §18.1 (Constraint Engine) is directly inspired by Draco's
constraint satisfaction framing, translated from ASP to explicit
Python rule evaluation against SQLviz's feature vector.

**How CBIOS goes further:**
Draco solves the single-chart selection problem.
CBIOS solves the dashboard-as-a-reasoning-system problem.

### Voyager (Wongsuphasawat et al., 2016, 2017)

**Paper:** "Voyager: Exploratory Analysis via Faceted Browsing of
Visualization Recommendations" (2016)
"Voyager 2: Augmenting Exploratory Analysis with Mixed-Initiative
Exploration" (2017)
**Origin:** University of Washington Interactive Data Lab

**What it does:**

```
Voyager is an exploratory analysis tool built on CompassQL.
It presents a gallery of recommended visualizations and allows
the user to "bookmark" interesting ones for further exploration.

Voyager 2 adds mixed initiative:
    User selects a chart → system suggests related charts
    System can also generate charts autonomously
```

**Key contribution:**
- First practical implementation of exploration-by-recommendation
- Mixed initiative: user and system share control of exploration
- Foundation for the "curiosity" concept CBIOS formalizes

**What Voyager does NOT do:**

```
❌ Operates on pre-loaded datasets, not SQL queries
❌ No analytical intent — treats all variables as symmetric
❌ No information gain — may recommend redundant charts
❌ No business domain understanding — field names are just strings
❌ No cognitive modeling, no uncertainty, no coverage tracking
❌ Recommendations are visual-structure driven, not insight driven
```

**How CBIOS goes further:**
Voyager's recommendation engine suggests charts that look different.
CBIOS's Curiosity Engine (Level 3) suggests questions that ADD
information — a strictly harder and more valuable problem.

### DeepEye (Luo et al., 2018)

**Paper:** "DeepEye: Towards Automatic Data Visualization"
**Origin:** Tsinghua University

**What it does:**

```
DeepEye uses a trained learning-to-rank model (pairwise comparison)
to score visualizations. Training data: human preferences collected
at scale.

Input:  Dataset + column set
Output: Top-ranked visualization

Two-stage pipeline:
    Stage 1: Rule-based pruning (remove clearly bad options)
    Stage 2: Learning-to-rank scoring (neural pairwise model)
```

**Key contribution:**
- First data-driven (not rule-based) visualization recommender
- Large-scale human preference dataset
- Shows that learned models outperform rule-based at scale

**What DeepEye does NOT do:**

```
❌ No SQL parsing — works on column sets
❌ No intent — treats all queries as "find best chart for columns"
❌ Black box: no explanations for recommendations
❌ No knowledge graph, no coverage, no information gain
❌ No user modeling beyond pairwise preference training
❌ Requires external training data not available in most organizations
```

**How CBIOS goes further:**
DeepEye learns to recommend charts that humans find visually appealing.
CBIOS learns to generate dashboards that maximize analytical value —
a different objective function entirely.

### VizML (Hu et al., 2019)

**Paper:** "VizML: A Machine Learning Approach to Visualization Recommendation"
**Origin:** MIT Media Lab

**What it does:**

```
VizML trains neural networks on 1M+ visualizations from Plotly
to predict five visualization design choices:
    - Is the visualization single-trace or multi-trace?
    - What mark type? (bar, line, scatter, ...)
    - Is the x-axis categorical or numerical?
    - Is the y-axis categorical or numerical?
    - Does the visualization use a colorscale?

Features: 841 statistical features extracted from the dataset
Model:    Neural network trained on 120K unique datasets
```

**Key contribution:**
- Largest-scale empirical study of visualization choices
- Shows that statistical features of data predict chart choices
- Validates the feature-vector approach SQLviz uses in V0.1

**What VizML does NOT do:**

```
❌ No SQL — works on pre-loaded tabular data
❌ No intent classification
❌ No explanations — neural network is a black box
❌ No dashboard reasoning — single chart only
❌ Requires Plotly corpus, not transferable to other domains
❌ No cognitive modeling, no coverage, no knowledge graph
```

**Connection to SQLviz:**
VizML validates the scientific premise of SQLviz's feature vector approach.
SQLviz's 39-dimension feature vector (DOC 5 §3) is a structured,
interpretable version of the same concept — with the key difference
that every dimension is explainable and every weight is adjustable.

### GraphScape (Kim et al., 2017)

**Paper:** "GraphScape: A Model for Automated Reasoning about
Visualization Similarity and Sequencing"
**Origin:** University of Washington Interactive Data Lab

**What it does:**

```
GraphScape models the "edit distance" between two visualizations
as a directed graph of elementary editing operations:

    Operations: change mark type, add/remove field, change encoding
    Cost:       each operation has a learned perceptual cost

Use cases:
    → Sequencing visualizations in a presentation
    → Suggesting "next visualization" in an exploration
    → Measuring how different two dashboards are
```

**Key contribution:**
- First formal model of visualization-to-visualization distance
- Enables reasoning about sequences of charts, not just individual ones
- Precursor to the "Dashboard Genome" concept in CBIOS Level 8

**What GraphScape does NOT do:**

```
❌ No SQL — operates on VegaLite specs
❌ Edit distance is structural, not information-theoretic
❌ No business domain understanding
❌ No cognitive model — structural distance ≠ cognitive distance
❌ No knowledge graph — no entity or metric tracking
```

**How CBIOS goes further:**
GraphScape measures how different two visualizations LOOK.
CBIOS Dashboard Genome (Level 8) measures how different two
dashboards ARE — in terms of information content, coverage, and
cognitive structure — which is the analytically relevant question.

**Origin:** MIT CSAIL

**What it does:**

**What Gleaner does NOT do:**

```
❌ No SQL — operates on pre-loaded datasets
❌ No business domain knowledge graph
❌ Interestingness is statistical, not business-meaningful
❌ No cognitive cost modeling
❌ No uncertainty quantification
❌ No coverage tracking
```

**How CBIOS goes further:**
Gleaner finds views that are statistically interesting.
CBIOS finds views that are analytically valuable — grounded in business
semantics, information gain, and the user's cognitive reasoning model.

## Difference from Current Commercial BI Tools

### The Rendering Paradigm (What Every BI Tool Does Today)

```
User intent
    → SQL or drag-and-drop
        → Rows of data
            → Chart configuration (manual)
                → Chart rendered
                    → Dashboard assembled (manual)
```

In this paradigm:

- The tool is a **renderer**, not a reasoner
- The dashboard is a **collection of charts**, not a reasoning system
- The user makes **every analytical decision** manually
- There is **no feedback loop** — the tool never learns from the user
- There is **no coverage** — the tool does not know what it does not know
- There is **no redundancy detection** — two panels can answer the same question
- There is **no cognitive modeling** — the tool does not model how humans read

### The Cognitive Compilation Paradigm (What CBIOS Does)

```
SQL query (the only input)
    → Knowledge Graph update
        → Information Gain calculation
            → Cognitive Cost optimization
                → Attention simulation
                    → Dashboard compiled (automatic)
                        → Curiosity Engine (next question)
                            → Coverage tracking
```

In this paradigm:

- The tool is a **cognitive compiler**, not a renderer
- The dashboard is a **compressed reasoning model**, not a chart collection
- The tool makes analytical decisions; the user **validates** them
- The feedback loop continuously improves output
- Coverage is explicit and measurable
- Redundancy is detected and eliminated
- Cognitive load is minimized by design

## The 10 Levels of the Cognitive BI Operating System

### Level 0 — The Current Paradigm and Why SQLviz Supersedes It

#### The SQL → Rows → Chart → Dashboard Paradigm

Every commercial BI tool operates on the same four-step pipeline:

```
Step 1 — SQL (or GUI query)
    User writes a query or drags fields

Step 2 — Rows
    Query executes → returns a table of rows

Step 3 — Chart (manual configuration)
    User selects: what chart? what X axis? what Y axis?
    what color? what filter? what title?

Step 4 — Dashboard (manual assembly)
    User drags charts into a grid
    User sets layout, sizes, titles, ordering
```

**The fundamental problem:** Steps 3 and 4 require the user to
make decisions that have nothing to do with the business problem
they are trying to solve. The cognitive effort spent configuring
charts is cognitive effort not spent analyzing data.

#### Why SQLviz Supersedes This Paradigm

```
V0.1 — SQL-First rendering:
    SQL → Inference Engine → Chart (automatic)
    → eliminates manual chart configuration

V0.2 — Cognitive Dashboard Compiler:
    SQL × N → Constraint Engine + Scoring + Layout Optimizer
    → eliminates manual layout configuration

V0.3+ (CBIOS) — Cognitive Compilation:
    SQL × N → Knowledge Graph → Information Gain
    → Coverage → Curiosity Engine → Compiled Dashboard
    → eliminates manual analytical decision-making

V1.0 — Cognitive Digital Twin:
    SQL × N × User → Cognitive Model → Personalized Compilation
    → the dashboard adapts to HOW the user reasons, not what they prefer
```

The progression from V0.1 to V1.0 is a systematic elimination
of every decision the user should not have to make.

### Level 1 — Knowledge Graph as Foundation

#### The Core Principle

A dashboard is not a collection of charts.
A dashboard is a VIEW of a Knowledge Graph.

Every SQL query is not just a data request.
Every SQL query is a **knowledge acquisition event** that enriches
the graph with new entities, metrics, and relationships.

#### The Business Knowledge Graph

```
BusinessKnowledgeGraph G = (E, M, R)

Where:
    E = set of business entities
        E = {Revenue, Orders, Customers, Products, Regions, Time, ...}

    M = set of metrics
        M = {SUM(revenue), COUNT(orders), AVG(ticket), ...}

    R = set of relationships between entities and metrics
        R = {(Revenue, measures, Orders),
             (Orders, has_dimension, Region),
             (Orders, has_dimension, Time),
             (Revenue, rolls_up_by, Month),
             (Revenue, rolls_up_by, Country)}
```

#### SQL as Knowledge Acquisition

Each SQL query enriches the graph:

```sql
-- Query 1
SELECT month, SUM(revenue) FROM orders GROUP BY month
```

```
Graph update:
    E += {Revenue, Time}
    M += {SUM(revenue)}
    R += {(Revenue, rolls_up_by, Month)}
    Coverage += {temporal_revenue_analysis: covered}
```

```sql
-- Query 2
SELECT country, SUM(revenue) FROM orders GROUP BY country
```

```
Graph update:
    E += {Geography}
    R += {(Revenue, rolls_up_by, Country)}
    Coverage += {geographic_revenue_analysis: covered}
```

```sql
-- Query 3
SELECT product, SUM(revenue) FROM orders GROUP BY product
```

```
Graph update:
    E += {Products}
    R += {(Revenue, rolls_up_by, Product)}
    Coverage += {product_revenue_analysis: covered}
    Note: Revenue entity already exists — no duplication
    Note: Time entity not involved — coverage gap remains
          on temporal × product cross-analysis
```

#### Dashboard as Graph View

The dashboard is not assembled. It is **compiled from the graph**.

```
Dashboard = project(G, user_context, cognitive_budget)

Where:
    G            = current state of the Business Knowledge Graph
    user_context = role, current analytical session, history
    cognitive_budget = available cognitive capacity (default: 7 chunks)

The compiler selects which subgraph to render,
what chart type maps to each subgraph component,
and how to order panels to maximize information flow.
```

#### Graph Persistence and Evolution

```
The graph is NOT reset between sessions.
Each SQL query executed anywhere in SQLviz enriches the graph.

Graph grows over time:
    Day 1:   Revenue × Time covered
    Week 1:  Revenue × Time × Geography × Product covered
    Month 1: Full revenue-cost-margin triangle covered
             Inventory-Sales relationship identified
             Customer-Revenue-Returns knowledge accumulated

The graph becomes the organization's analytical memory.
```

### Level 2 — Information Gain per Panel

#### The Core Principle

If two panels in a dashboard answer the same question,
one of them is waste. The user's cognitive budget is finite.
Redundant information consumes capacity without providing value.

SQLviz detects redundancy mathematically and eliminates it
before the dashboard reaches the user.

**This has never been implemented in any commercial BI tool.**

#### Formal Definition

For a set of panels P = {p₁, p₂, ..., pₙ} in a dashboard,
the information gain of adding panel pᵢ given the existing
set of panels P_prev = {p₁, ..., pᵢ₋₁} is:

```
IG(pᵢ | P_prev) = H(Q_target) - H(Q_target | P_prev ∪ {pᵢ})

Where:
    Q_target     = the analytical question the user is trying to answer
    H(Q_target)  = uncertainty about Q_target before any panels
    H(Q_target | ...) = remaining uncertainty after observing the panel set
```

For practical implementation in V0.3, approximated as:

```
IG(pᵢ | P_prev) = 1 - redundancy(pᵢ, P_prev)

redundancy(pᵢ, P_prev) = max_{pⱼ ∈ P_prev} similarity(pᵢ, pⱼ)

similarity(pᵢ, pⱼ) = f(
    semantic_overlap(pᵢ.entities, pⱼ.entities),
    metric_overlap(pᵢ.metrics, pⱼ.metrics),
    dimension_overlap(pᵢ.dimensions, pⱼ.dimensions)
)
```

#### Three Measurable Panel Properties

Every panel in the dashboard receives three computed scores:

```
information_gain  ∈ [0.0, 1.0]
    How much new information does this panel add?
    0.0 = completely redundant with another panel
    1.0 = completely unique, covers territory no other panel covers

redundancy        ∈ [0.0, 1.0]
    How much does this panel overlap with other panels?
    0.0 = no overlap
    1.0 = completely duplicated

novelty           ∈ [0.0, 1.0]
    How different is the analytical question from the user's
    typical session pattern?
    0.0 = user always asks this
    1.0 = user has never explored this territory before
```

#### Redundancy Detection Examples

```
Example 1 — Pure redundancy:

Panel A: SELECT month, SUM(revenue) FROM orders GROUP BY month
Panel B: SELECT year, month, SUM(revenue) FROM orders GROUP BY year, month

Analysis:
    entities(A) = {Revenue, Time}
    entities(B) = {Revenue, Time}
    metrics(A) = {SUM(revenue)}
    metrics(B) = {SUM(revenue)}
    semantic_overlap = 1.0

    IG(B | {A}) ≈ 0.05 — almost no new information
    redundancy(B, {A}) ≈ 0.95

    → SQLviz suggests: "Panel B duplicates Panel A at higher granularity.
      Consider combining into a single time-series with drill-down."

Example 2 — Partial redundancy:

Panel A: SELECT month, SUM(revenue) FROM orders GROUP BY month
Panel C: SELECT month, COUNT(orders) FROM orders GROUP BY month

Analysis:
    entities(A) = {Revenue, Time}
    entities(C) = {Orders, Time}
    metrics(A) = {SUM(revenue)}
    metrics(C) = {COUNT(orders)}
    metric_overlap = 0.0
    dimension_overlap = 1.0

    IG(C | {A}) ≈ 0.65 — significant new information (volume vs. revenue)
    redundancy(C, {A}) ≈ 0.35 — shared temporal dimension, different metric

    → SQLviz accepts Panel C: different metric, same time axis
      is analytically valuable (units sold vs. revenue earned)

Example 3 — Zero redundancy:

Panel A: SELECT month, SUM(revenue) FROM orders GROUP BY month
Panel D: SELECT category, AVG(return_rate) FROM returns GROUP BY category

Analysis:
    entities(A) = {Revenue, Time}
    entities(D) = {Returns, Products}
    All overlaps = 0.0

    IG(D | {A}) = 1.0 — completely new territory
    redundancy(D, {A}) = 0.0

    → SQLviz immediately accepts Panel D: covers analysis A does not reach
```

#### User-Facing Presentation

```

[Panel A]  ▓▓▓▓▓▓▓▓    Information Gain: 0.82
[Panel C]  ▓▓▓▓▓       Information Gain: 0.65
[Panel D]  ▓▓▓▓▓▓▓▓▓   Information Gain: 0.90

SQLviz recommendation:
    "Panel B adds minimal information. Panel A already covers revenue over time.

     or remove it entirely if you don't need monthly granularity."
```

### Level 3 — Curiosity Engine

#### The Core Principle

SQLviz does not wait for the next SQL query.

After each query execution, the Curiosity Engine analyzes
the current state of the Knowledge Graph, identifies analytical
gaps — entities and relationships NOT yet covered — and proposes
the next analytically valuable question.

The Curiosity Engine transforms SQLviz from a passive tool
into an **active analytical copilot**.

**This has no equivalent in any commercial BI tool.**

#### How the Curiosity Engine Works

```
State:  Current Business Knowledge Graph G
        Current dashboard panels P
        Current intent: "Revenue by month" → intent = "trend"

Analysis:
    G.covered = {Revenue × Time (temporal trend)}
    G.adjacent_uncovered = {
        "Revenue × Geography":   no query covers this
        "Revenue × Product":     no query covers this
        "Revenue decline cause": detected anomaly in trend data
    }

Output (ranked by analytical value):
    Suggestion 1: "Revenue dropped 22% in October.
                   → Investigate by: Region, Product, Sales Channel"
    Suggestion 2: "You've analyzed revenue over time.
                   → Which regions drive this revenue?"
    Suggestion 3: "October anomaly detected.
                   → Top 10 customers by month — has this client mix changed?"
```

#### Curiosity Trigger Taxonomy

The engine generates proposals based on five trigger types:

```
Trigger Type 1 — ANOMALY DETECTED
    Condition: data contains an anomaly (outlier, sudden change, break)
    Proposal:  "X changed — investigate cause by dimension D"
    Example:   "Revenue dropped 22% in October.
                Investigate by: Region / Product / Sales Channel"

Trigger Type 2 — UNCOVERED DIMENSION
    Condition: metric M is analyzed by dimension D1 but not D2
    Proposal:  "You know X by D1. Do you know X by D2?"
    Example:   "You know revenue by month. Do you know revenue by region?"

Trigger Type 3 — UNCOVERED RELATIONSHIP
    Condition: two entities A and B exist in the graph but are never
               analyzed together
    Proposal:  "Have you checked if A and B are correlated?"
    Example:   "You track both Returns and Revenue. Are they correlated?
                → Return rate by product category × revenue impact"

Trigger Type 4 — COVERAGE GAP
    Condition: a business domain area has coverage < threshold
    Proposal:  "Your dashboard has no visibility into [area]"
    Example:   "Your dashboard covers 95% of revenue analysis
                but only 8% of cost/loss analysis"

Trigger Type 5 — COMPARATIVE OPPORTUNITY
    Condition: user has established a baseline (current period)
               but no comparison (prior period, plan, benchmark)
    Proposal:  "You know the current state. What about the trend vs. plan?"
    Example:   "This shows October 2025 revenue.
                → Compare to October 2024 or annual target?"
```

#### Proposal Ranking

Proposals are ranked by a combined score:

```
proposal_score(s) = α × analytical_value(s)
                  + β × novelty(s)
                  + γ × feasibility(s)
                  - δ × cognitive_cost(s)

Where:
    analytical_value = estimated information gain if followed
    novelty          = distance from user's typical questions
    feasibility      = probability SQLviz can generate a good chart
    cognitive_cost   = estimated mental effort to evaluate the proposal

Weights (default):
    α = 0.40   (value matters most)
    β = 0.25   (some novelty is good)
    γ = 0.20   (only suggest what SQLviz can deliver)
    δ = 0.15   (low-cost proposals preferred)
```

#### Curiosity Engine as SQL Generator

At V1.0, the Curiosity Engine does not just propose a question.
It generates the SQL to answer it:

```
User accepts suggestion: "Which regions drive October's revenue drop?"

Curiosity Engine generates:
    SELECT
        region,
        SUM(CASE WHEN month = 10 AND year = 2025 THEN revenue ELSE 0 END) AS oct_2025,
        SUM(CASE WHEN month = 10 AND year = 2024 THEN revenue ELSE 0 END) AS oct_2024,
        SUM(CASE WHEN month = 10 AND year = 2025 THEN revenue ELSE 0 END) -
        SUM(CASE WHEN month = 10 AND year = 2024 THEN revenue ELSE 0 END) AS delta,
        ROUND(100.0 * (
            SUM(CASE WHEN month = 10 AND year = 2025 THEN revenue ELSE 0 END) -
            SUM(CASE WHEN month = 10 AND year = 2024 THEN revenue ELSE 0 END)
        ) / NULLIF(SUM(CASE WHEN month = 10 AND year = 2024 THEN revenue ELSE 0 END), 0), 2) AS pct_change
    FROM orders
    GROUP BY region
    ORDER BY delta ASC;
```

The user validates the SQL and executes it.
One click from analytical question to running query.

### Level 5 — Uncertainty Engine

#### The Core Principle

Every number on a dashboard has a confidence level.
The user should never trust a KPI blindly.
A trend based on 3 data points is not as reliable as a trend
based on 300 data points. SQLviz makes this explicit.

#### Uncertainty Sources

```
Source 1 — Sample size
    n < 10:     Low confidence (trend unreliable)
    n 10-100:   Medium confidence
    n > 100:    High confidence

Source 2 — Temporal recency
    Data is current month:  Full confidence
    Data is last quarter:   Slight decay
    Data is last year:      Significant decay (business may have changed)

Source 3 — Cardinality adequacy
    Group-by produces 2 values: trend inference is questionable
    Group-by produces 3+ values: trend inference is valid

Source 4 — Statistical noise
    High coefficient of variation (CV):  underlying data is noisy
    CV = std(values) / mean(values)
    CV > 0.5: noise flag

Source 5 — Coverage completeness
    Panel covers a domain with < 50% coverage: data may be incomplete
    Missing data in the underlying table: uncertainty from gaps
```

#### Panel Confidence Score

```
confidence(panel) = f(
    sample_size_score,
    recency_score,
    cardinality_score,
    noise_score,
    coverage_score
)

Computed as a weighted product:
    confidence = ∏ᵢ (weight_i × score_i)
    normalized to [0.0, 1.0]
```

#### User-Facing Confidence Display

```
[KPI Panel — Total Revenue]     Confidence: ████████████████████  99%
    "Based on 124,832 transactions. Data current to yesterday."

[Trend Panel — Monthly Revenue] Confidence: █████████████████░░░  84%
    "12 data points. Trend is statistically significant (p < 0.01)."

[New Markets Panel]             Confidence: █████░░░░░░░░░░░░░░░  28%
    "Only 3 data points. This trend may not be reliable.
     → Execute with more data before drawing conclusions."

[Return Rate by Segment]        Confidence: ██░░░░░░░░░░░░░░░░░░  12%
    "High data variability detected (CV = 0.84).
     Return rates vary significantly within each segment.
     → Consider narrower segments or longer time window."
```

### Level 6 — Cognitive Cost

#### The Core Principle

Every panel has a measurable cognitive cost.
The dashboard's total cognitive cost must not exceed the user's
available working memory capacity.

This is grounded in established cognitive science:
- Miller (1956): working memory capacity ≈ 7 ± 2 chunks
- Sweller (1988): Cognitive Load Theory — extraneous load reduces
  learning and decision quality
- Cowan (2001): actual working memory capacity ≈ 4 chunks

#### Panel Cognitive Cost Dimensions

```
CognitiveCost(panel) = f(
    reading_time,      # seconds to read and understand the panel
    chunk_count,       # distinct information chunks the panel contains
    information_density,  # bits of information per visual element
    processing_complexity  # number of comparisons required to understand
)
```

#### Cost by Chart Type (Baseline Estimates)

```
Chart Type          Chunk Count  Reading Time  Processing Complexity
──────────────────────────────────────────────────────────────────────
KPI (single number) 1            1.2s          0.1  (lowest cost)
Progress bar        1            1.5s          0.2
Line (1 series)     3            4.0s          0.4
Bar (< 6 bars)      4            3.5s          0.5
Bar (6-12 bars)     7            5.0s          0.7
Pie (< 5 slices)    4            4.5s          0.6
Scatter plot        6            8.0s          0.8
Heatmap             8            12.0s         0.9
Table (< 10 rows)   7            7.0s          0.7
Table (> 10 rows)   12           15.0s         0.95 (highest cost)
```

#### Dashboard Cognitive Budget

```
CognitiveBudget = 7 chunks (Miller's Law default)

Total cognitive cost of dashboard:
    TCC = Σᵢ CognitiveCost(pᵢ)

If TCC > CognitiveBudget:
    Dashboard is cognitively overloaded.
    SQLviz suggests:
    1. Remove the lowest information_gain panel
    2. Simplify the most complex panel
    3. Split into two dashboards by analytical theme
```

#### Cognitive Cost Optimization

The Layout Optimizer (V0.2, DOC 5 §18.5) is enhanced in V0.3
to include cognitive cost as an additional objective:

```
DashboardObjective_V0.3 = maximize(
    w₁ × information_coverage,
    w₂ × information_gain,
    w₃ × visual_clarity,
    -w₄ × cognitive_cost,      # minimize
    -w₅ × redundancy            # minimize
)
```

### Level 7 — Attention Simulation

#### The Core Principle

SQLviz knows where the user's eyes will go before the user opens the dashboard.

Users do not read dashboards linearly. They follow visual attention
patterns determined by:
- Salience: bright, large, contrast-heavy elements attract first
- Semantic priming: KPIs are processed before trends
- Working memory flow: once a number is processed, the eye seeks context

#### The Attention Model

Based on Healey & Enns (2012), Itti & Koch (2001) visual salience:

```
AttentionSequence(dashboard) = ordered list of panels in expected viewing order

Factors:
    1. Visual Salience:
       - Size: larger panels attract first
       - Color: high-contrast elements attract first
       - Position: top-left bias (Western reading pattern)
       - Anomaly: outliers in data attract first

    2. Semantic Pull:
       - KPIs are processed before charts (single numbers are low-cost)
       - Trends before comparisons (temporal narrative precedes comparison)
       - Summaries before detail (overview before zoom)

    3. Narrative Flow:
       - Introduction → Context → Detail → Conclusion
       - Standard analytical narrative (same as journalistic inverted pyramid)
```

#### Simulated Attention Output

```
Dashboard panels (as submitted):
    P1: Monthly Revenue (line chart, row 2)
    P2: Total Revenue KPI (single number, row 3)
    P3: Revenue by Region (bar chart, row 1)
    P4: Revenue by Product (table, row 4)

Simulated attention sequence:
    1st: P3 Revenue by Region (row 1 position + large chart)
    2nd: P2 Total Revenue KPI (high salience: single large number)
    3rd: P1 Monthly Revenue (trend context for P2)
    4th: P4 Revenue by Product (detail, processed last — if at all)

Problem detected:
    P4 (table) is in row 4. Simulated attention shows user
    does not reach P4 in typical session (< 40% reach rate).

SQLviz recommendation:
    "P4 is unlikely to be read. Simulated attention places it
     beyond the typical visual budget.
     → Consider: move P4 above the fold, or replace with a bar chart
     → Or: convert P4 to a searchable drill-down triggered from P3"
```

#### Attention Map Visualization

In V1.0, the Attention Map is displayed as a heat overlay on the dashboard:

```

████ High attention (≥80% of users reach here)
▓▓▓▓ Medium attention (40-80%)
░░░░ Low attention (<40% — consider redesigning)

    ████████████████ [Revenue KPI — 97%]
    ████████████████ [Monthly Revenue Line — 88%]
    ▓▓▓▓▓▓▓▓▓▓▓▓░░░ [Revenue by Region Bar — 61%]
```

### Level 8 — Dashboard Genome

#### The Core Principle

Every dashboard has a unique analytical DNA.
Two dashboards can be compared as precisely as two genomes.

The Dashboard Genome is a compact, structured descriptor of
the analytical structure of a dashboard — independent of the
specific data or time period.

#### Genome Structure

```python
@dataclass
class DashboardGenome:
    # Panel composition
    kpi_count: int                    # number of single-metric panels
    chart_count: int                  # number of chart panels
    table_count: int                  # number of table panels

    # Analytical narrative
    narrative_roles: list[str]        # ["trend", "comparison", "ranking", "diagnostic"]

    # Information metrics
    information_density: float        # total information gain / panel count
    business_coverage: float          # fraction of business domain covered
    redundancy_score: float           # average redundancy across panels
    uncertainty_avg: float            # average confidence across panels

    # Cognitive metrics
    total_cognitive_cost: float       # sum of panel cognitive costs
    complexity_score: float           # normalized complexity index [0,1]
    attention_efficiency: float       # fraction of panels in high-attention zone

    # Domain composition
    entity_coverage: dict[str, float] # {entity: coverage_fraction}
    metric_coverage: dict[str, float] # {metric: coverage_fraction}
    temporal_depth: str               # "daily"|"monthly"|"quarterly"|"annual"
```

#### Genome Distance

The distance between two dashboard genomes is:

```
d(G_A, G_B) = √(Σᵢ w_i × (G_A[i] - G_B[i])²)

Where i ranges over all numeric genome dimensions
and w_i is the importance weight of each dimension.
```

Use cases:

```

   "These two dashboards are 94% similar. Do you really need both?"

2. Dashboard templates:
   "Your new dashboard matches the 'Executive Revenue' template

    - 8 have low business coverage (< 0.4)
    - 3 have high cognitive cost (> 8 chunks)"

4. Cross-organization benchmarking:
   "Companies in your industry typically have executive dashboards
    with complexity_score < 0.3. Yours is 0.61."
```

### Level 9 — Dashboard Evolution

#### The Core Principle

A dashboard is not a static document.
A dashboard is a living organism that evolves as the user's
understanding deepens and the business changes.

SQLviz learns the user's analytical reasoning patterns and
evolves the dashboard accordingly.

#### Learning Dimensions

```
Dimension 1 — Chart type preference by intent:
    user corrects "bar" → "line" for trend intent 3 times
    → SQLviz learns: this user prefers line charts for trends
    → Weight update applied to PersonalScoringModel

Dimension 2 — Analytical sequence:
    user always opens dashboard, then:
        1. Checks KPIs (revenue, margin)
        2. Checks monthly trend
        3. Drills into regional breakdown
        4. Rarely reads product table
    → SQLviz learns the user's analytical narrative
    → Reorganizes panels to match this sequence

Dimension 3 — Domain focus:
    user consistently analyzes Revenue and Collection
    user never opens Inventory dashboards
    → SQLviz learns domain priority
    → Coverage recommendations focus on Revenue/Collection gaps

Dimension 4 — Time horizon preference:
    user always applies "last 3 months" filter manually
    → SQLviz learns: this user's preferred time horizon = 90 days
    → Applies 90-day default to new dashboards automatically

Dimension 5 — Question depth:
    user typically goes 2-3 drill-down levels
    user rarely goes to transaction-level detail
    → SQLviz learns: this user operates at aggregate level
    → Avoids recommending row-level detail panels
```

#### Behavioral Pattern Recognition

```
Financial Manager pattern:
    Revenue → Billing → Collection → Margin
    Typical session: 8 minutes, 4 dashboards, 2 drill-downs

Commercial Manager pattern:
    Sales → Products → Channels → Regions
    Typical session: 12 minutes, 6 dashboards, 4 drill-downs

Operations Manager pattern:
    Orders → Inventory → Delivery Time → Returns
    Typical session: 15 minutes, 8 dashboards, 6 drill-downs
```

SQLviz does not know these patterns in advance.
SQLviz DISCOVERS them through behavioral observation.
This is the foundation of the Cognitive Digital Twin.

#### Evolution Mechanism

```
Every user interaction generates a signal:
    → panel_viewed:      panel ID, timestamp, duration
    → chart_corrected:   old_type, new_type, context
    → filter_applied:    filter_type, value, frequency
    → panel_ignored:     panel ID, session_count_since_last_view
    → drill_down:        from panel A to query B
    → session_terminated: at panel P with context C

These signals feed the PersonalBehaviorModel:
    A Bayesian update model (see DOC 4 §Learning V1+)
    Updates PersonalScoringWeights continuously
    Decays old observations with half-life of 30 days
    (recent behavior matters more than historical)
```

### Level 10 — Cognitive Digital Twin

#### The Core Principle

SQLviz builds a model of HOW you think — not what you prefer.

Preference: "I like line charts."
Cognitive reasoning: "When I analyze revenue, I always ask:
                      What was the trend? → Then: What explains it?
                      → Then: What is the breakdown by region?
                      → Then: Which specific clients changed?"

The difference is profound.
Preference personalization gives the user charts they like.
Cognitive Twin personalization gives the user the analytical path
their mind was already going to follow — before they ask.

**This concept does not exist in any BI tool, academic or commercial.**

#### The Cognitive Digital Twin Model

```
CognitiveTwin(user) = {
    reasoning_template:   the user's typical analytical narrative
    entry_points:         which KPIs or domains the user opens first
    drill_pattern:        how many levels the user typically drills
    domain_sequence:      which business areas the user visits in order
    anomaly_sensitivity:  how sensitive the user is to unexpected data
    cognitive_capacity:   how many panels the user typically processes
    confidence_threshold: at what confidence level the user acts on data
    time_horizon:         the user's preferred analytical time window
}
```

#### Reasoning Template Example

```
Financial Manager — Cognitive Twin:

entry_points:      [Total Revenue KPI, Gross Margin KPI]
domain_sequence:   Revenue → Collection → Cost → Loss
drill_pattern:     entity → dimension (2 levels max)
anomaly_sensitivity: high (always investigates deviations > 5%)
cognitive_capacity: 6 panels per session
confidence_threshold: 0.85 (does not act on data below 85% confidence)
time_horizon:      rolling 90 days

Reasoning template:
    "Show me the number first.
     Then show me how it moved over time.
     Then show me who is responsible (region or channel).
     Then tell me if anything is unusual."

Dashboard compiled for this twin:
    Panel 1: Total Revenue KPI (current vs prior period)
    Panel 2: Gross Margin KPI (current vs prior period)
    Panel 3: Monthly Revenue Trend (12 months)
    Panel 4: Revenue by Region (bar, sorted descending)
    Panel 5: [Anomaly panel if detected, else: Revenue by Channel]
    Panel 6: [Cognitive budget exhausted — no Panel 6]

Total cognitive cost: 5.8 chunks (within budget of 6)
Attention efficiency: 0.94
```

#### Twin vs. Role-Based Personalization

```
Role-based (current industry approach):
    Role = "CFO" → apply CFO dashboard template
    → All CFOs get the same dashboard
    → Does not account for individual reasoning differences

Cognitive Twin (CBIOS):
    → Observe actual behavior of THIS user
    → Build a model of THIS user's reasoning
    → The CFO who focuses on cash flow gets a cash-flow-first dashboard
    → The CFO who focuses on margin gets a margin-first dashboard
    → Same role, different cognitive twin, different compiled dashboard
```

#### Twin Initialization

At first use, there is no behavioral data.
The twin initializes from:

1. Role selection (coarse prior): "Financial Manager" → default weights
2. Domain selection: "Which business areas do you track?" → coverage prior
3. First 5 sessions: rapid Bayesian update from initial behavior
4. After 30 sessions: twin is well-calibrated

#### Privacy and Transparency

```
The Cognitive Digital Twin is stored locally (SQLite).
No behavioral data leaves the user's machine.
The user can:
    → View their twin: "What has SQLviz learned about me?"
    → Reset their twin: "Forget everything, start over"
    → Edit their twin: "I don't actually care about channel analysis"
    → Export their twin: JSON format, portable across installations

Transparency principle (DOC 1, Principle 3):
    Every dashboard generated from a twin shows WHY:
    "This dashboard was ordered Revenue → Collection → Cost
     because that is your typical analytical sequence.
     [Override sequence] [Reset twin]"
```

## Mathematical Foundations for CBIOS

### Information-Theoretic Basis (Level 2)

```
Shannon Entropy of a panel's analytical question Q:
    H(Q) = -Σₓ p(x) log₂ p(x)

Where p(x) is the probability distribution over possible answers to Q.

Mutual Information between two panels p_i and p_j:
    MI(p_i; p_j) = H(p_i) - H(p_i | p_j)

Panel redundancy:
    redundancy(p_i, p_j) = MI(p_i; p_j) / H(p_i)
    ∈ [0, 1], where 1 = complete redundancy

Information gain of panel p_i given existing set P:
    IG(p_i | P) = H(p_i) - max_{p_j ∈ P} MI(p_i; p_j)
```

### Cognitive Load Basis (Level 6)

```
Sweller Cognitive Load Theory:
    Total Load = Intrinsic Load + Extraneous Load + Germane Load

For dashboard optimization:
    Intrinsic Load    = complexity inherent to the analytical question
    Extraneous Load   = complexity from poor visualization choice (minimized by V0.1-V0.2)
    Germane Load      = cognitive effort producing insight (what we want to maximize)

CBIOS objective:
    minimize Extraneous Load (via constraint satisfaction + scoring)
    minimize Intrinsic Load  (via panel decomposition + simplification)
    maximize Germane Load    (via curiosity + coverage recommendations)

Miller's Law:
    working_memory_capacity ≈ 7 ± 2 chunks
    → dashboard cognitive budget = 7 chunks (conservative default)
    → each panel's chunk_count must sum to ≤ 7
```

### Curiosity Engine Basis (Level 3)

```
Analytical Value of a proposal s:
    analytical_value(s) = E[IG(p_new | P)] if the user follows s

Where:
    p_new = the panel that would be generated if the user follows proposal s
    P = current panel set

This is the Expected Information Gain of following the proposal.
Higher expected gain = higher priority proposal.

Since we cannot observe the user's data before they execute the query,
we estimate E[IG] from:
    → The entities and metrics the proposal would cover
    → The current state of the Knowledge Graph
    → The historical IG of similar proposals in past sessions
```

## Glossary

```
CBIOS            Cognitive BI Operating System — the V0.3/V1.0 architecture
                 described in this document

Knowledge Graph  Graph of business entities (E), metrics (M), and their
                 relationships (R), constructed incrementally from SQL queries

Information Gain Per-panel measure of new analytical information added;
                 computed from mutual information between panel content

Redundancy       Per-panel measure of analytical overlap with other panels;
                 computed from entity + metric + dimension overlap

Coverage         Fraction of a business domain covered by the current
                 Knowledge Graph; computed per domain category

Curiosity Engine System that proposes the next analytical question based on
                 coverage gaps, anomalies, and uncovered relationships

Uncertainty      Per-panel confidence score computed from sample size,
                 data recency, noise level, and coverage completeness

Cognitive Cost   Measurable cognitive effort required to process a panel;
                 computed from chunk count, reading time, and complexity

Attention Model  Simulation of visual attention sequence on a dashboard;
                 based on salience (size, contrast, position) and semantics

Dashboard Genome Compact structured descriptor of a dashboard's analytical
                 DNA; enables cross-dashboard comparison and templating

Dashboard Twin   Behavioral model of an individual user's analytical
(Cognitive       reasoning pattern; built from interaction signals over time
Digital Twin)

SQL Event        A single SQL query execution treated as a knowledge
                 acquisition event that enriches the Knowledge Graph

Analytical       A typical reasoning sequence for a user: the ordered list
Narrative        of analytical questions they typically follow

Chunk            Miller's unit of working memory; approximately one
                 independently processable piece of information

IG               Information Gain (shorthand)
```
